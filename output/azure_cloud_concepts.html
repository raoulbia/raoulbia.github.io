Azure Cloud Concepts


Benefits of Cloud Services

High availability

  • Available for long periods of time
  • availability is NOT backup or disaster recovery
  • availability is NOT high performance
  • availability is NOT load balancing

Scalability vs. Elasticity

  • Scalability: increase/decrease workload on current resources
  • Elasticity: increase/decrease the number of resources e.g. adding/removing VMs

Agility

  • Quickly respond to changes
  • Provision or decommission cloud services in minutes using self-service

Disaster recovery

  • Ability to recover from a failure e.g. backups for VMs or SQL databases

Capital Expenses (CapEx) vs. Operational Expenses (OpEx)

  • CapEx are predictable upfront costs
  • OpEx are ongoing billing costs (pay as you go) and can be deducted from your taxes immediately.
  • OpEx suited to agile environments where demand is unpredictable

Consumption-based model

  • Paying for something based on how much you used, as opposed to paying for something no matter if you use it or not.

Categories of Cloud Services

IaaS

  • get a VM and do what you want with it
  • emulation of a real piece of HW that you can buy in a store
  • includes virtualised representation of networking, load balancers, firewall
  • 1-to-1 mapping/moving of what you have on-premises, e.g. pieces of hardware, to the cloud
  • does NOT include SW services e.g. DB's
  • you control the HW, choice of datacentre, availability zone, nbr of CPU’s
  • VM’s, Azure Storage, Load balancer, VPNs, network gateways, subnets ; tie things together , move things around
  • You are responsible for upgrades, patches, version numbers

PaaS

  • in Azure they are called Web Apps or App Services
  • It’s a platform that you need to build upon to be useful
  • development effort is required to turn it into sth useful
  • You pay by the hour even if no one visits your App or website
  • Cloud provider is more involved in this paradigm
  • You compile code/program, upload it and cloud provider runs that code
  • not picking out specific pieces of HW, not setting up networks; no choices over the HW
  • You do have to choose tiers, and plan levels within the tiers (see e.g. App Service plan) but not RAM, CPU’s or manage disk space
  • You pay for performance (the tiers)
    • you have some idea of how many instances are running your code, and what your performance expectations are
    • scaling is your responsibility: you must decide whether to increase nbr of CPU’s
  • it is programmable so that you can upload your own code (App Service)
  • or it has to be programmable so that it can integrate with other things (Azure SQL Database, Cosmos DB, Machine Learning Studio)
  • It is more useful than raw networking or raw computing

SaaS

  • Dropbox, backup services, google drive, google docs, office365, outlook, Active Directory, CRM tool, any enterprise App,
  • Anything where you use a web browser to access a service
  • Access to configuration of existing product only; no access to code
  • Don’t control servers; only control data and settings
  • The product is useful stand alone

Serverless

  • Name is misleading; There are still servers, you just don’t have to ever deal with them
  • no access to server ; don’t know how many servers or processors are running your code
  • no control over performance expectations (as opposed to performance via tiers and plans in PaaS)
  • Cloud provider is responsible for scaling e.g. adding servers or CPU’s as they are needed
  • Don’t have to worry about choosing the right plan or scaling (up or down)
  • Means you might pay nothing if you don’t use the service i.e. no one ever visits your App.
  • You only pay for execution and not for reservation
  • You pay for CPU and data utilisation, NOT visits
  • available serverless offers:
    • Compute – Azure functions
    • Compute – serverless kubernetes (virtual nodes w/ACI
    • Database – Azure SQL Database serverless
    • Database – Cosmos DB serverless

Shared Responsibility model

Types of Cloud Computing

  • Public

    • anyone can sign up for it
    • azure owns the HW on their network and infrastructure ; you are leasing services

  • Private:

    • you use your own datacentre, HW, upfront capital expenditure (CapEx)
    • you can license software from the Azure Stack, install on your own hardware
    • you can use a portal, create VMS etc just like you do with the public cloud
    • You own the HW etc but you use the services of the public cloud thus getting benefits of flexibility, agility and elasticity
    • You don’t get the cost savings by just stopping to run sth

  • Hybrid:

    • extending your disk space or your on-premises databases into the public cloud
    • i.e. scaling private infrastructure to the cloud
